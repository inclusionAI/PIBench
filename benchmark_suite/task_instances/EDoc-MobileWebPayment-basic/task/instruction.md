# 任务：为 eDoc 患者预约流程补齐移动浏览器支付宝支付能力

## 背景

你面前是一个名为 **eDoc** 的医生预约系统（PHP + MySQL/MariaDB）。患者可以登录、查看医生排班、选择预约时间并提交预约。当前系统可以创建预约记录，但还缺少一条适合手机浏览器内跳转支付宝完成付款的链路。

代码位于你的工作目录（`/workspace`）。请直接在该目录内修改 / 新增 PHP 文件来完成任务。

## 任务目标

为患者预约流程补齐移动浏览器支付宝付款能力：患者提交预约后，系统应创建**待支付**预约单，生成可跳转到支付宝收银台的付款入口；支付完成后，预约**只能在服务端可信确认后**变为已支付 / 已确认。

## 具体要求

### 后端服务

1. 保留现有患者登录、排班、预约页面，不要新增一套替代预约系统。
2. 预约提交后应先进入待支付状态，再进入支付流程；不要在创建预约时直接确认成功。
3. 后端需要提供创建支付宝手机网站支付请求和接收服务端支付结果的能力，前端入口和后端接口保持一致。
4. 支付结果以服务端可信确认为准；前端回跳或用户退出只能用于展示当前状态，不能单独决定预约成功或失败。
5. 支付成功后再把预约推进到已支付 / 已确认，并保持重复通知或重复访问时的结果稳定。
6. 支付配置应来自环境变量或等价运行时配置，不要把真实密钥提交进源码。
7. 站内支付入口可以先落在本系统页面，但移动端最终应基于运行时支付宝配置生成手机网站支付请求，并进入支付宝沙箱收银台链路。

### 固定接口约定（评测以这些 endpoint 为准）

| 方法 | 路径 | 期望行为 |
| --- | --- | --- |
| `GET`  | `/login.php` | 登录页可访问。 |
| `POST` | `/login.php` | 患者账号可登录（成功后 302/303 跳转）。 |
| `GET`  | `/patient/schedule.php` | 查看可预约排班。 |
| `GET`  | `/patient/booking.php?id={schedule_id}` | 预约确认页。 |
| `POST` | `/patient/booking-complete.php` | 创建**待支付**预约单，并 302/303 跳转到包含 `out_trade_no=...` 的支付流程。 |
| `POST` | `/patient/alipay-h5/payment.php` | 创建移动网页支付入口，返回 **JSON**，至少包含 `payment_url`、`out_trade_no`、`amount`、`currency`、`payment_status`。 |
| `GET`  | `/patient/alipay-h5/pay.php?out_trade_no=...` | 移动端跳转支付入口；桌面端（`?desktop=1`）可展示便于复制或继续在手机上打开的支付入口。 |
| `GET`  | `/patient/alipay-h5/return.php?out_trade_no=...` | 用户前端回跳页；只能展示或触发服务端确认后的状态，不应直接确认支付成功。 |
| `GET`  | `/patient/alipay-h5/quit.php?out_trade_no=...` | 用户中途退出页；只能展示当前状态，不应直接把预约改成失败或取消。 |
| `POST` | `/alipay/h5/notify.php` | 接收服务端支付通知：有效通知可信确认后推进预约状态；无效或订单金额不匹配的通知应拒绝，重复有效通知保持幂等。 |

### 数据与金额约定

- 预约金额为 `99.00`，币种 `CNY`（数据库 `appointment` 表已预置相关列；也提供了 `appointment_payment` 表用于记录支付请求）。
- `payment.php` 返回的 `payment_url` 应绑定当前 `out_trade_no`，并能继续进入站内或支付宝手机网站支付请求；本地中转路径按项目现有路由风格实现，前后端保持一致即可。
- 本地验收会用支付宝通知常见字段模拟服务端支付结果；处理时应校验订单号、金额、支付状态和签名/可信来源，金额必须与订单一致才可入账。

### 健康检查

- `GET /health.php` 返回 200，JSON 中 `ok=true`（已实现，无需改动，但你的改动不能让它退化）。

### 运行时配置

验收环境会提供支付宝沙箱配置。请优先从 `ALIPAY_APP_ID`、`ALIPAY_GATEWAY` / `ALIPAY_GATEWAY_URL`、`ALIPAY_PRIVATE_KEY` / `ALIPAY_MERCHANT_PRIVATE_KEY`、`ALIPAY_NOTIFY_URL`、`ALIPAY_RETURN_URL`、`ALIPAY_QUIT_URL` 等运行时配置读取；不要把真实密钥或固定外部地址写死在源码里。本地异步通知 fixture 的可信签名值会通过 `ALIPAY_NOTIFY_FIXTURE_VALID_SIGN` / `ALIPAY_SANDBOX_VALID_SIGN` 提供，只应用在本地验收 fixture 里，不要写成脱离环境控制的生产绕过。

## 边界说明

- 不要求实现退款、部分退款或资金安全加固（属于后续 q2-safety）。
- 不要求接入生产密钥；验收会提供沙箱运行时配置，并用本地可信 fixture 模拟异步支付通知。
- 不要求新增完整后台管理 UI。

## 运行环境提示

- 应用以 `php -S` 内置服务器方式提供服务，数据库为本机 MariaDB；数据库连接参数由 `connection.php` 从环境变量读取（`DB_HOST`/`DB_PORT`/`DB_NAME`/`DB_USER`/`DB_PASSWORD`）。
- 你新增的 endpoint 必须落在上面约定的固定路径上，评测脚本会以 HTTP 方式访问这些路径。
- 完成后无需自己启动服务或运行测试，评测会自动构建数据库、启动服务并打分。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
