# 任务：加固 eDoc 医生预约系统的支付宝支付链路

## 背景

你面前是一个已经具备基础移动浏览器支付宝付款链路的 **eDoc** 医生预约系统（PHP 8 + MariaDB）。
患者可以创建待支付预约单，系统可以生成移动网页付款入口，也能接收支付结果。当前实现里有若干
**资金安全 / 资金正确性** 行为还不够可靠。

## 任务目标

请对这条支付链路做安全和资金正确性加固。重点是让预约履约、支付状态、退款结果和本地资金记录都以
服务端可信结果和本地状态机为准，避免前端状态、外部回调、重复请求或退款边界处理不当造成错误履约或资损。

## 工作目录

项目代码在你当前工作目录（`/workspace`）下，是一个标准 PHP 站点。你应当直接修改其中的 PHP 文件。
应用通过 `connection.php` 读取数据库连接环境变量，数据库为 MariaDB。

建议先梳理支付相关的公共库、患者侧 H5 支付入口、异步通知入口、后台退款入口和运行环境说明。
请沿用项目现有结构做加固，不要另起一套绕过原流程的实现。

## 具体要求

### 后端服务

1. 保留现有患者预约、支付和后台退款入口，不要新增一套替代 API。
2. 支付状态变更应遵循清晰的服务端状态机。前台回跳、用户退出、客户端参数或接口调用成功码，不能单独触发履约、取消或资金变更。
3. 支付通知和查单补偿应校验可信来源、订单绑定、金额、商户身份和支付状态，并且对重复请求保持幂等。
4. 同一待支付预约重复发起支付时，应复用或确认已有支付请求，避免产生第二笔可支付交易。
5. 已支付预约应支持退款和部分退款；退款处理需要结合本地订单状态、可退金额、外部退款结果和退款请求标识，避免越权退款、未支付退款或重复扣减。
6. 已支付、部分退款、全额退款等终态需要受到保护，不能被旧通知、失败通知、退出流程或不明确的外部状态覆盖。
7. 支付和退款配置应来自运行时环境或项目已有配置，不要提交真实密钥或敏感 `.env`。

### 固定接口约定（不可更改方法与路径）

| 方法 | 路径 | 期望行为 |
| --- | --- | --- |
| `POST` | `/patient/alipay-h5/payment.php` | 创建或复用待支付预约的支付入口。 |
| `GET` | `/patient/alipay-h5/pay.php?out_trade_no=...` | 移动端跳转支付入口，桌面端展示可复制链接。 |
| `GET` | `/patient/alipay-h5/return.php?out_trade_no=...` | 前台回跳页，展示或触发服务端确认后的状态。 |
| `GET` | `/patient/alipay-h5/quit.php?out_trade_no=...` | 用户退出页，展示当前支付状态。 |
| `POST` | `/patient/alipay-h5/sync.php` | 对待确认支付执行服务端查单补偿。 |
| `POST` | `/alipay/h5/notify.php` | 接收并处理支付宝服务端支付通知。 |
| `POST` | `/admin/alipay-h5/refund.php` | 对符合条件的已支付预约发起退款或部分退款。 |
| `POST` | `/admin/alipay-h5/refund-query.php` | 查询并回写退款处理中或未知结果。 |

### 边界说明

- 不要求接入生产密钥；可以使用沙箱、mock 服务或本地 fixture（仓库默认 `mock_mode`）。
- 不要求新增完整后台管理 UI。
- 不要求删除已有预约流程。

## 约定与提示

- 数据库连接相关环境变量与 `connection.php` 已配置好，运行时由评测环境注入；你无需修改连接逻辑，
  也不要把数据库主机、端口写死。
- 本地 mock/fixture 环境会提供可验证的成功和失败样例；验签逻辑应走项目已有的 mock 开关或运行时配置，
  不要把测试值做成脱离环境控制的生产绕过。
- 金额口径请保持区分：应付、实付、已退和可退金额是不同概念。部分退款、退款查询和重试处理不要混用单一金额字段。
- 不要提交真实密钥或敏感 `.env`。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
