# litemall 电商网站支付安全加固

## 背景

你面前是一个 litemall Java 电商系统，已经具备 PC 网站支付的基础入口：用户可以发起支付宝电脑网站支付，系统也已有支付通知入口和订单状态流转雏形。但当前实现还不足以支撑生产级资金安全，异常路径、重复请求和退款边界都需要进一步加固。

## 任务目标

请在现有工程结构中加固这条支付宝电脑网站支付链路。系统必须只接受可信的支付宝支付结果，支付通知和本地订单必须能够相互匹配；重复通知、异常状态、同步跳转、客户端参数或不确定查询结果都不能导致错误履约、重复履约或状态覆盖。退款、交易查询和关单也需要符合真实资金状态边界，避免重复退款、超额退款、错误关单或把未知结果当成成功。

## 接口与配置约定

为避免接口猜测，后端需要提供以下可调用契约，内部 service 分层和命名可以自行设计：

- `POST /wx/order/alipay-query`：请求体包含 `orderId`，服务端以支付宝交易查询结果确认支付状态；只有支付宝明确返回支付成功且金额与本地订单一致时才能履约，待支付、未知或异常结果不能入账。
- `POST /wx/order/alipay-refund`：请求体包含 `orderId`、退款金额（如 `amount`/`refundAmount`/`refund_amount`）和退款请求号（如 `outRequestNo`/`out_request_no`）。必须校验订单已支付、单次和累计可退金额、退款请求号幂等；同一次重试保持相同 `out_request_no`，不同部分退款使用不同 `out_request_no`，累计超额必须在调用支付宝退款网关前拒绝，`fund_change=N` 不能当作最终退款成功。
- `POST /wx/order/alipay-close`：请求体包含 `orderId`，只允许关闭未支付订单；已支付订单不能关单或覆盖状态，应保持已支付或走退款路径。

项目中提供了支付宝沙箱配置材料和现有支付代码，请以适合本地/容器运行的方式完成后端加固。App ID、网关地址和回调地址可以通过配置或环境变量接入；完整私钥/公钥等密钥材料不能硬编码在业务源码或提交的配置默认值中，本地敏感配置也应避免提交到版本控制。

## 边界说明

本题关注 PC 网站支付的资金安全和订单状态正确性，不要求实现营销、分账、对账、监控或复杂运营后台能力。前端只需要保持现有支付入口可用，核心改动应集中在后端支付结果确认、退款、查询和订单状态保护上。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
