# 任务：EZ Tickets 支付宝支付安全加固

## 背景

你面前是一个已经具备支付宝 App 支付基础能力的电影票务系统。用户可以在移动端创建票务支付并唤起支付宝，但当前实现还不足以支撑真实资金链路中的安全确认、异常补偿和退款边界。

## 任务目标

请把这条链路加固为资金结果可信的实现：客户端返回只能作为线索，最终支付、退款和履约状态必须由服务端通过可信通知或查单确认。系统需要保证支付结果与本地订单、用户、金额、交易号和收款方身份一致；未知、处理中、重复请求、旧通知和旧查询结果不能造成错误履约、重复退款或终态覆盖。

## 工作目录

项目位于 `/workspace`：

- `ez_tickets_backend/`：Node.js 20 + Express 4 + MySQL 8 后端，是主要改造对象。
- `ez_tickets_app/`：Flutter 客户端，可按需调整客户端支付返回后的调用逻辑。
- `alipay-sandbox-keys.json`：支付宝沙箱测试 fixture 密钥，非生产密钥。
- `start.sh`：启动本机 MySQL、导入 `ez_tickets.sql` 并拉起后端的脚本。

## 改造目标

请在现有工程结构中完成后端安全加固，保持原有电影列表、选座/购票和支付创建流程可用。支付创建、客户端确认、服务端查单、异步通知、退款和本地订单状态之间应形成一致的业务关系。

系统应以运行期支付宝配置完成支付、查单、验签和退款请求。测试环境会通过环境变量注入本地 mock 支付宝网关和测试密钥；生产密钥、固定网关、固定成功响应或绕过验签逻辑都不应写入业务代码。

退款逻辑需要尊重真实资金状态和订单归属：只有归属用户的已支付订单才允许进入退款流程；退款接口调用成功不等于资金已经变化；重复退款请求、部分退款、超额退款和退款查询不确定态都应有清晰处理，不能把不确定结果直接当成最终成功。

## 固定接口契约

评测会直接调用以下固定入口。可以调整内部实现，但不要新增一套替代 API 来绕过这些入口。

| 方法 | 路径 | 要求 |
| --- | --- | --- |
| `GET` | `/api/v1/movies` | 无需登录，返回 200 和电影列表。 |
| `POST` | `/api/v1/payments/alipay/create` | 登录态；创建支付宝 App 支付请求，返回可供客户端拉起支付的参数和业务支付标识。 |
| `GET` | `/api/v1/payments/alipay/:out_trade_no` | 登录态；查询当前支付状态、金额和关联 booking 信息。 |
| `POST` | `/api/v1/payments/alipay/:out_trade_no/confirm` | 登录态；处理客户端从支付宝返回后的确认请求，只能触发服务端确认流程，不能仅凭客户端结果完成履约。 |
| `POST` | `/api/v1/payments/alipay/:out_trade_no/sync` | 登录态；对 pending/unknown 支付执行服务端查单补偿，只有可信成功结果才能推进支付成功。 |
| `POST` | `/api/v1/payments/alipay/:out_trade_no/refund` | 登录态；仅允许订单归属用户对已支付订单发起退款，支持必要的幂等重试和部分退款处理。 |
| `POST` | `/api/v1/payments/alipay/notify` | 无需登录；支付宝异步通知入口，只有可信且能匹配本地订单的通知才能推进状态。 |

## 配置约束

支付宝配置优先从环境变量读取，缺省时可回退到仓库根目录的 `alipay-sandbox-keys.json` 作为本地测试 fixture。涉及的运行期配置包括 `ALIPAY_APP_ID`、`ALIPAY_SELLER_ID`、`ALIPAY_PRIVATE_KEY`、`ALIPAY_PUBLIC_KEY`、`ALIPAY_GATEWAY`、`ALIPAY_NOTIFY_URL`、`ALIPAY_SIGN_TYPE` 等。

不要把真实密钥、生产证书、固定成功结果或测试专用白名单写入业务源码。若新增本地配置文件，应确保敏感配置不会被提交。

## 边界说明

- 不要新增复杂营销后台或与票务支付无关的功能。
- 不要接入真实生产密钥。
- 不要通过写死测试通知、固定成功查询、无条件验签成功、客户端回调直接 paid、跳过退款调用等方式通过测试。

## 如何运行与自测

项目根有 `start.sh`，可在已装好 MySQL 的环境里启动后端：

```bash
cd /workspace
./start.sh
curl -i http://127.0.0.1:3331/api/v1/health
curl -i http://127.0.0.1:3331/api/v1/movies
```

后端依赖未安装时先在 `ez_tickets_backend/` 中安装依赖。请确保你的改动后后端仍能正常启动，并且上述固定接口可访问。

## 交付

直接在 `/workspace/ez_tickets_backend/` 中修改代码，必要时可调整 `/workspace/ez_tickets_app/`。完成后无需提交 PR，评测会自动收集改动、构建并运行测试。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
