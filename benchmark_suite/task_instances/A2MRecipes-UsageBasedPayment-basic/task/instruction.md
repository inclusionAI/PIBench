# 任务：为 A2M Recipes 接入付费资源访问

## 背景

你面前是一个名为 **A2M Recipes** 的宝宝辅食食谱服务（Next.js 16 + TypeScript，位于当前工作目录）。
它面向 AI Agent 提供食谱查询能力，当前已有食谱列表和食谱详情接口，但完整食谱详情还没有接入付费访问控制。

## 任务目标

请在智能体场景接入支付宝支付，应该叫AI收，使公开食谱列表仍可访问，而完整食谱详情需要支付凭证；服务端确认凭证有效后，才返回完整食材和制作步骤。

## 业务要求

1. 保留现有食谱接口，不要新增一套替代 API 让原流程失效。
2. 食谱列表应继续公开访问；完整食谱详情应作为付费资源保护起来。
3. 未携带支付凭证时，服务端应返回可供 Agent 发起支付的结构化挑战信息，并且不能泄漏完整食材和步骤。
   该挑战信息应足以让调用方发起支付，能够表达当前付费资源、支付金额、币种、应用或商家相关标识，以及本次支付请求标识，并适合放入 HTTP header 被客户端解析。
4. 携带支付凭证时，服务端应通过运行时配置的支付网关进行服务端确认，并把验证和当前资源、本次支付请求关联起来；只有网关明确确认凭证有效后，才返回完整内容。
5. 网关不可达、超时、返回异常、明确拒绝，或只表示接口调用成功但没有明确支付/凭证有效状态时，应按无法确认支付处理，不得放行付费资源。
6. 网关确认成功后通常会返回用于交付确认的信息；完整付费资源实际返回给用户后，服务端应把这段信息带回支付网关，完成交付或履约确认。
7. 支付相关配置应来自运行时环境或等价配置，不要把真实密钥、证书或 token 写进源码。

## 接口与配置约定

- `GET /api/recipes`：公开食谱列表，不应要求支付。
- `GET /api/recipes/:id`：完整食谱详情，需要支付凭证。
- 支付凭证通过 `Payment-Proof` request header 传入。
- 未完成支付时，通过 `Payment-Needed` response header 返回支付挑战信息。
- `Payment-Needed` 应以适合 HTTP header 传输的可解析结构返回，避免因编码或字符集问题导致接口不可用。
- 支付配置契约以项目根目录预置的 `.env.example` 为准。评测环境会按照其中声明的固定变量名注入本地支付网关和测试凭据；实现应读取这些名称，可以兼容额外别名，但不要删除、重命名或只支持其他未声明名称。
- 运行时网关地址必须来自该配置，不要连接写死的生产或真实沙箱网关。测试密钥不得写入源码、日志或客户端响应；配置缺失、无效或网关不可用时不得释放付费资源。

## 边界说明

- 不要求实现退款、部分退款或营销优惠。
- 不要求新增后台管理 UI。
- 不要求接入真实生产密钥；评测环境会提供仅用于本地 fixture 的测试值。

## 验收方式说明

- 评测系统会安装依赖、构建项目，并启动服务进行 HTTP 集成测试。
- 你不需要让服务保持运行，完成代码修改即可。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
