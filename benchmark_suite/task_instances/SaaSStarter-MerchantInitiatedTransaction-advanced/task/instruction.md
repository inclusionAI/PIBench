# SaaS Starter：商家扣款订阅安全加固

## 背景

SaaS Starter 已经有一个最小的支付宝商家扣款订阅 POC，但它还没有处理真实订阅业务里最容易导致资损或状态错乱的边界。本题真正考察的是签约、扣款、通知和状态机的安全加固。

## 接口与配置约定

评测侧会启动 Python 版支付宝 mock 网关，并通过 `ALIPAY_GATEWAY` 或 `ALIPAY_GATEWAY_URL` 注入地址。请使用这个外部网关，不要连接生产或真实沙箱网关，也不要在业务应用里加入直接改状态的测试捷径。真实私钥、token、app secret 和生产凭据都不能写入业务源码、数据库或日志。

## 任务目标

业务上需要把签约和扣款分开建模。签约请求通过 `alipay.trade.app.pay` 创建，并绑定团队、用户、套餐、`external_agreement_no` 和签约订单；扣款通过 `alipay.trade.pay` 使用已保存协议发起，并绑定当前账期、金额和 `out_trade_no`。网关受理、pending、处理中、超时或同步成功样返回都不能直接激活付费权益，最终成功应来自验签后的通知或权威查询。

通知处理需要先验签，再校验应用/商户身份、协议号、用户身份、团队归属、订单号、金额和状态等关键字段。重复通知、旧通知、上一笔扣款未确认时的新扣款、超过协议金额或周期限制的扣款，都不能造成重复收费、重复激活或覆盖终态。解约后应停止后续扣款；未知状态和网关异常应保持 fail-closed，并通过协议查询或交易查询补偿。

为了让评测稳定观察你的实现，请保留这些应用入口，内部组织方式不限：`POST /api/alipay/sign-contract`、`POST /api/alipay/sign-notify`、`POST /api/alipay/withhold`、`POST /api/alipay/pay-notify`、`GET /api/alipay/status?teamId=1`。这些入口需要真实驱动签约记录、扣款记录、订阅状态和网关调用。

## 完成说明

完成后请概述你如何验证正常签约扣款、异常通知、幂等、pending 处理、查询补偿、解约停止和密钥管理。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
