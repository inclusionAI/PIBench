# 任务：A2M Proof 付费资源交付安全加固

## 背景

你面前是一个已经完成基础 AI 收（支付宝 A2M / 智能收）接入的宝宝辅食食谱服务
（Next.js 16 + TypeScript，包管理使用 pnpm）。服务可以在用户未支付时返回支付要求，
也会在收到 `Payment-Proof` 后尝试返回付费食谱详情。

当前实现只能跑通基础流程，但还不能作为真实付费资源交付链路使用。请你把它加固成一个
服务端可信的支付验证与资源交付流程：完整付费内容只能在服务端确认支付凭证、支付结果、
资源关系和交付流程都可信之后返回。

## 工作目录与范围

- 代码在当前工作目录（项目根目录）下。
- 你可以修改、新增项目内的源码、配置和测试文件，但**只在本项目目录内操作**。
- 包管理只用 `pnpm`。如需安装依赖，使用 `pnpm install`。

## 业务安全目标

这个接口面向付费食谱资源。`ingredients` / `steps` 属于完整付费内容，不能因为客户端传了
一个看起来像 proof 的 header 就直接返回。你需要让服务端完成必要的校验、状态判断和交付
保护，使付费内容不会被伪造凭证、错配交易、失效凭证、重复请求或异常外部状态错误解锁。

实现时请保持以下业务语义：

1. 未携带支付凭证时，服务端继续返回支付要求，而不是完整食谱详情。
2. 服务端不能信任客户端传入的支付状态、金额、资源标识或确认结果。
3. 只有服务端确认支付凭证属于当前请求的付费资源，并且支付与交付都处于可信成功路径时，才返回完整详情。
4. 无法确认、状态异常、重复请求、外部服务异常、确认失败或凭证失效时，应拒绝交付或保持待支付状态。
5. 不要为了本地测试方便加入跳过验证、固定白名单 proof、无条件成功或泄漏密钥的逻辑。

## 接口与配置约定

- `GET /api/recipes`：无需支付 header，返回食谱摘要列表。
- `GET /api/recipes/:id`：付费食谱详情接口。
- 未支付或不能可信确认支付时，详情接口不能返回完整 `ingredients` / `steps`。
- 需要支付时，响应应包含 `Payment-Needed` header。
- 用户完成支付后，会通过 `Payment-Proof` header 携带支付凭证访问详情接口。

支付配置契约以项目根目录预置的 `.env.example` 为准。评测环境会按照其中声明的固定变量名
注入本地支付网关和测试凭据；实现应读取这些名称，可以兼容额外别名，但不要删除、重命名
或只支持其他未声明名称。运行时网关地址必须来自该配置，不要连接写死的生产或真实沙箱网关。
测试密钥不得写入源码、日志或客户端响应；配置缺失、无效或网关不可用时不得释放付费资源。

集成测试会命中上面的 HTTP 入口、`Payment-Proof` / `Payment-Needed` header 名称和关键响应行为。
不要重命名这些路由或 header，否则测试无法命中你的实现。

## 边界说明

- 不要求实现退款、部分退款或营销优惠。
- 不要求新增后台管理 UI。
- 不要求真实生产密钥；测试环境会提供可控的支付验证/履约确认端点。可以使用测试配置，
  但不能用无条件本地成功、固定白名单 proof 或跳过 verify/confirm 的方式替代服务端确认链路。
- **不要把任何真实私钥、证书或 token 写进仓库**。

## 完成与验收说明

- 服务仍可通过 `pnpm install && pnpm build && pnpm start` 构建并启动；
- `GET /api/recipes` 与无凭证的 `GET /api/recipes/:id` 行为符合上面的固定接口约定；
- 支付凭证校验与完整资源交付逻辑仍然挂在 `GET /api/recipes/:id` 这条链路上；
- 不可信、错配、失效、重复或异常状态不会返回完整付费食谱内容。

完成任务前，请停止并清理你在本次任务中启动的所有应用服务、后台进程和 Docker/Compose 容器，并确认相关监听端口已经释放。
